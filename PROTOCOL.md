## Notes & caveats

- **Format:** plain UTF-8 JSON objects sent as WebSocket text frames (opcode 1). No SignalR/protobuf/binary framing observed.
- **Discriminator:** `PID`. Observed values: `(no PID)`, `LTS_TIMESYNC`, `0`, `3`, `4`, `7`, `9002`.
- **Stringified numbers:** the server encodes most numeric values (lap times, sector lengths, speeds, IDs) as JSON strings. The `VER` field is observed as both a number and a string across different PIDs — likely a serialization quirk on the server.
- **Empty-string sentinels:** unset numeric fields (e.g. `S7TIME`) come back as `""`, not `null`. Treat empty strings as "not available".
- **Stationary RESULT cardinality:** every PID=`"0"` snapshot observed in this sample carried exactly 161 entries in `RESULT`. That is likely the full entry list; track it for changes once cars retire.
- **Time-sync round trip:** the only client→server frame in this capture is `{eventId, eventPid, clientLocalTime}`; the server echoes it back as a `PID=LTS_TIMESYNC` message with an added `serverLocalTime`.
- **Sample is short:** a 30-second window during qualifying only surfaced 3 distinct PIDs. Expect more PIDs (incident messages, sector-time updates, pit events) once you capture during the race itself — re-run the analyzer over the full capture.

## PID = 7 — two subscription forms

The server emits PID=7 in two operating modes that the same client can drive
in parallel:

| Form         | Subscribe payload                                                              | Behaviour                                                      |
| ------------ | ------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| Live stream  | `{eventId, eventPid:[7], clientLocalTime}` (combined with other PIDs is fine)  | Emits new TYPE=0 frames as cars complete laps; **no backfill** |
| Lap history  | `{eventId, eventPid:[7], clientLocalTime, json:{session, startingNo:""}}`      | Emits one cumulative TYPE=0 frame per car covering L=1..N, then keeps the socket open for new laps |

- `session` for the lap-history form comes from any prior PID=0 or PID=4
  frame's `SESSION` field — capture once and reuse.
- `startingNo` is ignored by the server (verified live: parallel subscriptions
  with different car numbers returned identical streams). Send empty.
- `capture.ts` opens a second WS in the lap-history form once it sees a
  session id and tags the resulting frames `source: "lapHistory"` in the
  JSONL. Both connections appear in the per-connection summary above.

### TYPE = 0 — per-car cumulative lap history

Each TYPE=0 frame carries one car's full lap list, sorted by `L`. **Each frame
replaces** the car's lap list — do not append-and-dedupe. Top-level fields:

| Field      | Meaning                                                          |
| ---------- | ---------------------------------------------------------------- |
| `EXPORTID` | event id (string, matches `eventId` in the subscribe)            |
| `HEATTYPE` | heat-type code (`"T"` for timed sessions)                        |
| `SESSION`  | session id matching the subscribe envelope (lap-history form)    |
| `SECTORS`  | intermediates count, e.g. `"9"`                                  |
| `N`        | car start number                                                 |
| `DATA`     | `LtsLap[]` — the car's full lap list                             |

DATA element shape:

| Field          | Meaning                                                          |
| -------------- | ---------------------------------------------------------------- |
| `L`            | lap number (1-based, monotonic per car)                          |
| `N`            | car start number (matches the outer `N`)                         |
| `D`            | wall-clock completion timestamp, epoch ms                        |
| `T`            | total lap time string — see below                                 |
| `S1`…`S9`      | sector times as `M:SS.mmm` strings (cumulative from line cross)  |
| `V1`…`V9`      | speed-trap reading at each intermediate, km/h (integer)          |

#### What is `T`?

Hypotheses considered:

1. Pit-stop time on a pit lap.
2. Total elapsed since session start at the moment this lap closed.
3. Delta from a reference (best lap / class leader).
4. Continuation marker (always zero for valid laps).
5. Day-time HH:MM:SS rendered as M:SS.fff due to a string bug.

The 4-minute race capture (`race-30min.jsonl`, ≈ 34 MB, with both the live
and lap-history forms running in parallel) disproved all of these except a
sixth, simpler definition:

> **`T` = `D(L) − D(L−1)` exactly**, formatted as `M:SS.mmm` /
> `MM:SS.mmm` / `H:MM:SS.mmm`. Identical semantics across both forms.

Across the capture:

- 250 / 250 lap pairs from the **live (page)** stream matched `D(L) − D(L−1)`
  within rounding tolerance.
- 250 / 250 lap pairs from the **lapHistory** stream matched the same rule.
- Only 44 / 250 happened to also match "elapsed since L=1" — that's
  coincidence for L=2 (where the two definitions are identical) and small
  numbers of later in-pit-lap edge cases; from L=3 onward the two diverge.
- `T = "00.000"` occurs in exactly two situations:
  - **First lap delivered for a car**: no prior `D` to subtract from.
  - **Preliminary / in-progress entry with `D = 0`**: lap not yet finalized.

So `T` is the **server-side authoritative lap time**, derived from the
between-line-crossings delta of cumulative timestamps. It's redundant with
`Σ(S1..S9)` for laps where no intermediate is dropped; the value of having
it on the wire is that a client which misses one sector can still get a
correct lap total.

The hypothesis from the original write-up — that the example value
`"16:31.397"` (lap 2 for car 80) represented session-elapsed time — turns
out to be coincidence: that same value is also `D(2) − D(1)` for that car,
because for `L=2` "since session start" and "since previous crossing" are
the same expression.

Pit / slow-zone laps surface as outliers in `T` (the 14:08, 16:09 and
similar values in the sample reflect cars sitting in code-60 zones or in
the pit lane for part of a lap), but `T` itself is not "pit time" — it
includes both the lap and any time spent stationary in pit. To flag pit
laps, compare `T` (or `Σ(S1..S9)`) against the car's median; anything
≥ ~15 % above median is a pit / slow-zone lap.

The dashboard's `lapTotalMs()` helper now uses `Σ(S1..S9)` as the
authoritative lap-duration value (it's always exactly equal to a non-zero
`T` and is more robust when only a partial lap is present). The Lap Chart
applies the `> median × 1.15` rule for pit-flag colouring.

### TYPE = 1 — one-shot class-best progression

Delivered shortly after subscribe; inner `DATA` is an array of
`{CLASS, LAPTIME, S1}` entries — one per class-leader best lap so far in
the session. Treat as a one-time bootstrap.

## PID = 9002 — class statistics

Initial push is large (~188 KB); subsequent frames are incremental updates
to one or more of the three top-level arrays:

- `LEADING[]` — per-class leader stints. Each entry has `{CLASS, NR,
  FROMLAP, LAPS, SUM}` where `NR` is the leading car's start number,
  `FROMLAP` is the lap on which it took the lead, and `LAPS`/`SUM` count
  laps led.
- `BESTLAPS[]` — `{CLASS, NR, INLAP, LAPTIME, DAYTIME}` — class-best lap
  with the car number that set it, the lap number on which it was set, the
  lap time, and the wall-clock time it was set.
- `BESTSECTORS[]` — `{CLASS, LAPTIME, S1..S9}` — theoretical-best per-class
  sector chain.

Useful for "class leader stints", "best lap of class with car and lap
number", and theoretical-best sector aggregates.
